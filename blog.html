Markov Chain Music Generation

A Markov chain for music is a lookup table: given the last n notes played, sample the next note according to how often each transition appeared in the training data. The "order" of the chain is how many previous notes constitute the state.

Training

Training scans the input sequence and counts every n-gram transition. For a first-order chain on "Twinkle Twinkle," you end up with a table like: 

transitions["[60]"] = { 60: 2, 67: 4, 69: 2 }
// from C4: go to C4 (25%), G4 (50%), or A4 (25%)

Generation starts from a random known state and repeatedly samples the next note, shifting the state window forward each step. When a dead-end state is hit (no outgoing transitions), the implementation falls back to a random known state.

Orders

Order 1 output sounds reasonable but has no tangible structure. In other words, each note follows naturally from the one before, but there's no sense of direction. Order 2 or 3 sounds noticeably more structured. The chain still has no model of phrases or keys, but a larger context window is enough to make it more pleasing to the ear. Higher order also means more training data is required. With only about 42 notes in Twinkle Twinkle, a 4th-order chain has so many unique states that most only appear once.

Difficulties

I found audio scheduling to be the most difficult part of this process, as Web Audio API notes must be queued ahead of time via AudioContext.currentTime, not standard time. Making the "stop" button work correctly was also difficult given that all oscillators are scheduled upfront, so you have to explicitly call .stop() on each one to cut playback immediately.

Conclusion

The transition matrix visualization reveals that the chain learns the tonal structure of the input purely from statistics. As the chain has no built-in notion of scales or keys, the patterns in the matrix come entirely from the training data.
